# 客户与联系人 Excel 导入模板设计

日期：2026-05-19

## 背景

当前系统是 Fishpool 客户池管理系统，后端使用 Django REST Framework，前端使用 Vue 3 + Element Plus。系统已经具备客户 CRUD、联系人 CRUD、客户导出、全量工作簿导出，以及营收数据 Excel/CSV 导入能力。

本次需求是增加“客户导入”能力，并把需要导入的客户字段和联系人字段做成一个 Excel 模板文件放在项目文件夹里，方便用户按模板批量维护客户与联系人数据。

## 目标

1. 提供一份客户与联系人导入 Excel 模板文件，放在项目仓库的固定目录中。
2. 模板只使用一个工作表，一行同时包含客户字段和联系人字段。
3. 支持用户在客户列表页上传 Excel 文件批量导入客户和联系人。
4. 客户字段与现有“新增客户”表单保持一致。
5. 联系人字段与现有联系人模型保持一致。
6. 客户等级不在 Excel 中手工填写，由系统根据 X/Y/Z 评分自动计算。
7. 导入后返回客户新增、客户更新、联系人新增、联系人更新、跳过和错误明细，方便用户检查数据质量。

## 不做范围

1. 不导入营收数据，营收已有独立导入入口。
2. 不支持一个联系人同时绑定多个客户。
3. 不做联系人模糊匹配或去重预览。
4. 不做异步任务队列，当前数据量按同步导入处理。
5. 不引入新的 Excel 库，继续使用现有 `openpyxl`。

## 模板文件位置

模板文件建议放在：

```text
docs/templates/customer_import_template.xlsx
```

理由：

1. 这是给业务用户使用的文件，不是 Django 模板，因此不放在 `backend/templates/`。
2. `docs/templates/` 语义清晰，后续如果有 Weekly Report、营收等模板，也可以集中管理。
3. 后端下载接口可以直接读取该文件返回给浏览器。

## Excel 工作表结构

模板使用一个 `.xlsx` 工作簿，只有一个工作表：

```text
客户导入
```

第一行是固定表头。从第二行开始，每一行代表一个“客户 + 联系人”组合：

1. 如果客户不存在，则创建客户。
2. 如果客户已存在，则更新客户。
3. 如果该行填写了联系人姓名，则创建或更新该客户下的联系人。
4. 如果该行没有填写联系人姓名，则只导入或更新客户，不处理联系人。

一个客户有多个联系人时，在多行中重复填写同一个客户名称和客户字段，再分别填写不同联系人信息。

## 模板字段

`客户导入` 工作表第一行是固定表头。字段如下：

| 列名 | 是否必填 | 对应系统字段 | 说明 |
| --- | --- | --- | --- |
| 客户名称 | 是 | `client_name` / `name` | 用于新增或匹配已有客户 |
| 业务模式 | 是 | `business_model` | 仅允许 `Hunting` 或 `Farming` |
| 区域 | 否 | `area` / `region` | 对应客户区域 |
| 城市 | 否 | `city` | 对应客户城市 |
| 地址 | 否 | `address` | 详细地址 |
| X轴描述 | 否 | `description_x` / `score_x_desc` | 客户潜力说明 |
| X轴评分 | 否 | `score_x` | 0-100 数字；为空时按 0 导入 |
| Y轴描述 | 否 | `description_y` / `score_y_desc` | 竞争环境说明 |
| Y轴评分 | 否 | `score_y` | 0-100 数字；为空时按 0 导入 |
| Z轴描述 | 否 | `key_person` / `score_z_desc` | 关键人关系说明 |
| Z轴评分 | 否 | `score_z` | 0-100 数字；为空时按 0 导入 |
| 客户策略 | 否 | `client_strategy` / `strategy` | 客户策略 |
| 潜在贡献 | 否 | `potential_contribution` | 数字，单位沿用系统现有定义 |
| 备注 | 否 | `remark` / `notes` | 备注信息 |
| 联系人姓名 | 否 | `contacts.name` | 填写后才导入联系人 |
| 联系人职位 | 否 | `contacts.position` | 联系人职位 |
| 联系人电话 | 否 | `contacts.phone` | 联系电话 |
| 联系人邮箱 | 否 | `contacts.email` | 邮箱地址 |
| 是否关键人 | 否 | `contacts.is_key_person` | 支持 `是/否`、`Y/N`、`true/false`、`1/0` |

模板不包含“客户等级”列。客户等级按现有规则自动计算：

1. Level A：X、Y、Z 都大于等于 60。
2. Level B：X、Y、Z 中至少两项大于等于 60。
3. Level C：X 大于等于 60。
4. Level D：其他情况。

## 导入规则

### 客户匹配规则

以“客户名称”做精确匹配：

1. 如果系统中存在同名客户，则更新该客户。
2. 如果系统中不存在同名客户，则创建新客户。
3. 客户名称导入前后去除首尾空格。

先采用精确匹配，避免模糊匹配误更新错误客户。后续如果业务需要，可以再单独设计“模糊匹配预览”。

### 联系人匹配规则

联系人由同一行中的联系人字段导入：

1. 如果“联系人姓名”为空，则该行不处理联系人。
2. 如果“联系人姓名”不为空，则联系人绑定到该行匹配到的客户。
3. 以“客户 + 联系人姓名”做精确匹配。
4. 如果该客户下存在同名联系人，则更新联系人。
5. 如果该客户下不存在同名联系人，则创建联系人。

该规则允许不同客户下存在同名联系人，也避免同一客户下重复创建同名联系人。

### 重复客户行处理

因为单 Sheet 需要支持一个客户多个联系人，同一个客户可能在多行出现。处理规则如下：

1. 同一个客户第一次出现时，校验并保存客户字段。
2. 同一个客户后续再次出现时，只处理联系人字段，不再覆盖客户字段。
3. 如果后续同名客户行的客户字段与第一次出现不一致，返回 warning，提示已保留第一次出现的客户字段。
4. 用户仍应尽量保证同一客户多行中的客户字段保持一致。
5. 联系人字段按每行分别处理。

该规则可以支持一个客户多个联系人，同时避免某一行复制错误导致客户主数据被意外覆盖。

### 更新规则

Excel 中每一行作为一条客户记录和最多一条联系人记录处理。更新已有记录时，模板中的字段会覆盖系统中对应字段。

空值处理规则：

1. 客户必填字段为空：跳过整行并记录错误，联系人也不处理。
2. 客户非必填字段为空：更新为空值。
3. 潜在贡献为空：保存为空值。
4. 联系人姓名为空：不处理联系人，不计为错误。
5. 联系人姓名不为空但其他联系人字段为空：对应联系人字段更新为空值。
6. 联系人“是否关键人”为空：保存为 `false`。

## 数据校验

导入时需要校验：

1. 文件类型必须是 `.xlsx`。
2. 工作簿必须包含 `客户导入` 工作表，或默认解析第一个工作表。
3. 第一行必须包含模板表头。
4. 客户名称、业务模式不能为空。
5. 业务模式必须是 `Hunting` 或 `Farming`。
6. X/Y/Z 评分为空时按 0 导入；如果填写，必须是 0-100 的数字。
7. 潜在贡献如果填写，必须是数字。
8. 联系人姓名为空时，不校验联系人其他字段。
9. 联系人姓名不为空时，“是否关键人”如果填写，必须能解析为布尔值。
10. 完全空白的数据行跳过，不计入错误。

## 后端设计

在 `CustomerViewSet` 上新增两个 action：

1. `GET /api/customers/import-template/`
   - 返回 `docs/templates/customer_import_template.xlsx`。
   - 下载文件名建议为 `customer_import_template.xlsx`。

2. `POST /api/customers/import/`
   - 使用 `multipart/form-data` 上传文件，字段名为 `file`。
   - 解析 `客户导入` 工作表；如果找不到该名称，则解析第一个工作表。
   - 按表头读取每一行。
   - 先校验并新增或更新该行客户。
   - 如果该行填写了联系人姓名，再新增或更新该联系人。
   - 返回导入结果。

返回结构建议：

```json
{
  "customers_created": 3,
  "customers_updated": 5,
  "contacts_created": 8,
  "contacts_updated": 2,
  "skipped": 2,
  "errors": [
    {
      "row": 4,
      "customer_name": "示例客户",
      "contact_name": "张三",
      "reason": "是否关键人必须是 是/否、Y/N、true/false 或 1/0"
    }
  ]
}
```

后端可以新增小型工具函数，避免把导入解析逻辑全部堆在 action 内：

1. `_read_customer_import_rows(upload)`
2. `_parse_required_text(row, column)`
3. `_parse_score(value, column)`
4. `_parse_optional_decimal(value, column)`
5. `_parse_boolean(value, column)`
6. `_calculate_customer_level(score_x, score_y, score_z)`
7. `_create_customer_import_template()`
8. `_import_customer_and_contact_rows(rows)`

## 前端设计

在客户列表页现有工具栏中增加两个按钮：

1. “下载模板”
   - 调用 `GET /api/customers/import-template/`。
   - 下载 `customer_import_template.xlsx`。

2. “导入客户”
   - 打开隐藏的文件选择框。
   - 只接受 `.xlsx`。
   - 调用 `POST /api/customers/import/`。
   - 导入成功后刷新客户列表。
   - 用消息提示展示客户新增、客户更新、联系人新增、联系人更新、跳过数量。

前端 API 文件 `frontend/src/api/customer.js` 新增：

1. `downloadCustomerImportTemplate()`
2. `importCustomers(file)`

## 错误处理

后端返回 400 的场景：

1. 未上传文件。
2. 文件后缀不是 `.xlsx`。
3. 文件无法被 `openpyxl` 读取。
4. 缺少模板表头。

单行数据错误不终止整个导入，只跳过该行并加入 `errors`。为了避免响应过大，错误列表最多返回前 50 条。

前端展示策略：

1. 整体文件错误：直接弹出错误消息。
2. 部分行错误：显示“导入完成：客户新增 X，客户更新 Y，联系人新增 A，联系人更新 B，跳过 Z”。
3. 如果需要查看具体错误，后续可再增加导入结果弹窗；本次先返回数据并在控制台保留完整响应。

## 测试计划

后端测试：

1. 下载模板接口返回 `.xlsx`，且包含一个工作表和完整表头。
2. 上传模板格式的 Excel，能创建新客户。
3. 上传同名客户，能更新已有客户。
4. 评分导入后自动计算客户等级。
5. 空白 X/Y/Z 评分会按 0 导入，并参与客户等级计算。
6. 同一行填写联系人姓名时，能为客户创建联系人。
7. 同一客户下同名联系人再次导入时能更新联系人。
8. 联系人姓名为空时，只导入客户，不创建联系人。
9. 同一客户多行不同联系人时，能导入多个联系人。
10. 缺少客户必填字段时跳过整行并返回错误。
11. 非法业务模式、非法评分、非法潜在贡献、非法布尔值会跳过对应行。
12. 完全空白行会被忽略。

前端验证：

1. 客户列表页能下载模板。
2. 客户列表页能选择 `.xlsx` 文件上传。
3. 导入完成后客户列表刷新。
4. 上传失败时显示后端返回的错误信息。

## 待确认点

1. 更新已有客户或联系人时，Excel 中的空白非必填字段是否允许覆盖系统原值为空。当前设计是允许覆盖。
2. 客户名称是否必须精确匹配才更新客户。当前设计是精确匹配，避免误更新。
3. 同一客户下同名联系人是否视为同一联系人并更新。当前设计是更新。
4. 同一客户多行出现时，后面的客户字段是否允许覆盖前面的客户字段。当前设计是不覆盖，并返回 warning。
5. 导入结果错误明细本次是否只返回给接口，不做前端弹窗详情。当前设计是先不做详情弹窗。
