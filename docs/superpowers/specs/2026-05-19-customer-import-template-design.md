# 客户与联系人 Excel 导入模板设计

日期：2026-05-19

## 背景

当前系统是 Fishpool 客户池管理系统，后端使用 Django REST Framework，前端使用 Vue 3 + Element Plus。系统已经具备客户 CRUD、联系人 CRUD、客户导出、全量工作簿导出，以及营收数据 Excel/CSV 导入能力。

本次需求是增加“客户导入”能力，并把需要导入的客户字段和联系人字段做成一个 Excel 模板文件放在项目文件夹里，方便用户按模板批量维护客户与联系人数据。

## 目标

1. 提供一份客户与联系人导入 Excel 模板文件，放在项目仓库的固定目录中。
2. 支持用户在客户列表页上传 Excel 文件批量导入客户和联系人。
3. 客户字段与现有“新增客户”表单保持一致。
4. 联系人字段与现有联系人模型保持一致。
5. 客户等级不在 Excel 中手工填写，由系统根据 X/Y/Z 评分自动计算。
6. 导入后返回客户新增、客户更新、联系人新增、联系人更新、跳过和错误明细，方便用户检查数据质量。

## 不做范围

1. 不导入营收数据，营收已有独立导入入口。
2. 不支持一个联系人同时绑定多个客户。
3. 不做联系人模糊匹配或去重预览。
4. 不做异步任务队列，当前数据量按同步导入处理。
5. 不引入新的 Excel 库，继续使用现有 `openpyxl`。

## 模板文件位置

模板文件建议放在：

```text
docs/templates/customer_import_template.xlsx
```

理由：

1. 这是给业务用户使用的文件，不是 Django 模板，因此不放在 `backend/templates/`。
2. `docs/templates/` 语义清晰，后续如果有 Weekly Report、营收等模板，也可以集中管理。
3. 后端下载接口可以直接读取该文件返回给浏览器。

## Excel 工作表结构

模板使用一个 `.xlsx` 工作簿，包含两个工作表：

1. `客户`
2. `联系人`

导入顺序固定为先导入 `客户` 工作表，再导入 `联系人` 工作表。这样联系人可以通过“客户名称”关联到刚刚新增或已经存在的客户。

如果工作簿缺少 `联系人` 工作表，本次设计建议视为格式错误，因为用户明确需要导入联系信息。这样可以避免用户误用只有客户字段的旧模板。

## 客户工作表字段

`客户` 工作表第一行是固定表头。字段如下：

| 列名 | 是否必填 | 对应系统字段 | 说明 |
| --- | --- | --- | --- |
| 客户名称 | 是 | `client_name` / `name` | 用于新增或匹配已有客户 |
| 业务模式 | 是 | `business_model` | 仅允许 `Hunting` 或 `Farming` |
| 区域 | 是 | `area` / `region` | 对应客户区域 |
| 城市 | 是 | `city` | 对应客户城市 |
| 地址 | 否 | `address` | 详细地址 |
| X轴描述 | 否 | `description_x` / `score_x_desc` | 客户潜力说明 |
| X轴评分 | 是 | `score_x` | 0-100 数字 |
| Y轴描述 | 否 | `description_y` / `score_y_desc` | 竞争环境说明 |
| Y轴评分 | 是 | `score_y` | 0-100 数字 |
| Z轴描述 | 否 | `key_person` / `score_z_desc` | 关键人关系说明 |
| Z轴评分 | 是 | `score_z` | 0-100 数字 |
| 客户策略 | 否 | `client_strategy` / `strategy` | 客户策略 |
| 潜在贡献 | 否 | `potential_contribution` | 数字，单位沿用系统现有定义 |
| 备注 | 否 | `remark` / `notes` | 备注信息 |

模板不包含“客户等级”列。客户等级按现有规则自动计算：

1. Level A：X、Y、Z 都大于等于 60。
2. Level B：X、Y、Z 中至少两项大于等于 60。
3. Level C：X 大于等于 60。
4. Level D：其他情况。

## 联系人工作表字段

`联系人` 工作表第一行是固定表头。字段如下：

| 列名 | 是否必填 | 对应系统字段 | 说明 |
| --- | --- | --- | --- |
| 客户名称 | 是 | `customer.client_name` | 用于关联客户，必须能精确匹配客户 |
| 联系人姓名 | 是 | `name` | 联系人姓名 |
| 职位 | 否 | `position` | 联系人职位 |
| 电话 | 否 | `phone` | 联系电话 |
| 邮箱 | 否 | `email` | 邮箱地址 |
| 是否关键人 | 否 | `is_key_person` | 支持 `是/否`、`Y/N`、`true/false`、`1/0` |

联系人工作表允许一个客户出现多行，用于导入多个联系人。

## 导入规则

### 客户匹配规则

以 `客户` 工作表中的“客户名称”做精确匹配：

1. 如果系统中存在同名客户，则更新该客户。
2. 如果系统中不存在同名客户，则创建新客户。
3. 客户名称导入前后去除首尾空格。

先采用精确匹配，避免模糊匹配误更新错误客户。后续如果业务需要，可以再单独设计“模糊匹配预览”。

### 联系人匹配规则

联系人通过 `联系人` 工作表中的“客户名称”关联客户：

1. 先在本次导入成功的客户和系统已有客户中查找同名客户。
2. 找不到客户时，跳过该联系人行并记录错误。
3. 找到客户后，以“客户 + 联系人姓名”做精确匹配。
4. 如果该客户下存在同名联系人，则更新联系人。
5. 如果该客户下不存在同名联系人，则创建联系人。

该规则允许不同客户下存在同名联系人，也避免同一客户下重复创建同名联系人。

### 更新规则

Excel 中每一行作为一条客户或联系人记录处理。更新已有记录时，模板中的字段会覆盖系统中对应字段。

空值处理规则：

1. 必填字段为空：跳过该行并记录错误。
2. 非必填字段为空：更新为空值。
3. 潜在贡献为空：保存为空值。
4. 联系人“是否关键人”为空：保存为 `false`。

### 数据校验

导入时需要校验：

1. 文件类型必须是 `.xlsx`。
2. 工作簿必须包含 `客户` 和 `联系人` 两个工作表。
3. 两个工作表第一行必须包含对应模板表头。
4. 客户名称、业务模式、区域、城市、X/Y/Z 评分不能为空。
5. 业务模式必须是 `Hunting` 或 `Farming`。
6. X/Y/Z 评分必须是 0-100 的数字。
7. 潜在贡献如果填写，必须是数字。
8. 联系人姓名不能为空。
9. 联系人“是否关键人”如果填写，必须能解析为布尔值。
10. 完全空白的数据行跳过，不计入错误。

## 后端设计

在 `CustomerViewSet` 上新增两个 action：

1. `GET /api/customers/import-template/`
   - 返回 `docs/templates/customer_import_template.xlsx`。
   - 下载文件名建议为 `customer_import_template.xlsx`。

2. `POST /api/customers/import/`
   - 使用 `multipart/form-data` 上传文件，字段名为 `file`。
   - 解析 `客户` 和 `联系人` 两个 Sheet。
   - 先校验并新增或更新客户。
   - 再校验并新增或更新联系人。
   - 返回导入结果。

返回结构建议：

```json
{
  "customers_created": 3,
  "customers_updated": 5,
  "contacts_created": 8,
  "contacts_updated": 2,
  "skipped": 2,
  "errors": [
    {
      "sheet": "联系人",
      "row": 4,
      "customer_name": "示例客户",
      "reason": "联系人姓名不能为空"
    }
  ]
}
```

后端可以新增小型工具函数，避免把导入解析逻辑全部堆在 action 内：

1. `_read_workbook_sheet_rows(workbook, sheet_name, required_columns)`
2. `_parse_required_text(row, column)`
3. `_parse_score(value, column)`
4. `_parse_optional_decimal(value, column)`
5. `_parse_boolean(value, column)`
6. `_calculate_customer_level(score_x, score_y, score_z)`
7. `_create_customer_import_template()`
8. `_import_customer_rows(rows)`
9. `_import_contact_rows(rows)`

## 前端设计

在客户列表页现有工具栏中增加两个按钮：

1. “下载模板”
   - 调用 `GET /api/customers/import-template/`。
   - 下载 `customer_import_template.xlsx`。

2. “导入客户”
   - 打开隐藏的文件选择框。
   - 只接受 `.xlsx`。
   - 调用 `POST /api/customers/import/`。
   - 导入成功后刷新客户列表。
   - 用消息提示展示客户新增、客户更新、联系人新增、联系人更新、跳过数量。

前端 API 文件 `frontend/src/api/customer.js` 新增：

1. `downloadCustomerImportTemplate()`
2. `importCustomers(file)`

## 错误处理

后端返回 400 的场景：

1. 未上传文件。
2. 文件后缀不是 `.xlsx`。
3. 文件无法被 `openpyxl` 读取。
4. 缺少 `客户` 或 `联系人` 工作表。
5. 缺少任一工作表的模板表头。

单行数据错误不终止整个导入，只跳过该行并加入 `errors`。为了避免响应过大，错误列表最多返回前 50 条。

前端展示策略：

1. 整体文件错误：直接弹出错误消息。
2. 部分行错误：显示“导入完成：客户新增 X，客户更新 Y，联系人新增 A，联系人更新 B，跳过 Z”。
3. 如果需要查看具体错误，后续可再增加导入结果弹窗；本次先返回数据并在控制台保留完整响应。

## 测试计划

后端测试：

1. 下载模板接口返回 `.xlsx`，且包含 `客户` 和 `联系人` 两个 Sheet。
2. 模板中两个 Sheet 的表头与设计字段一致。
3. 上传模板格式的 Excel，能创建新客户。
4. 上传同名客户，能更新已有客户。
5. 评分导入后自动计算客户等级。
6. 上传联系人 Sheet，能为客户创建联系人。
7. 同一客户下同名联系人再次导入时能更新联系人。
8. 联系人关联不到客户时跳过该行并返回错误。
9. 缺少必填字段时跳过该行并返回错误。
10. 非法业务模式、非法评分、非法潜在贡献、非法布尔值会跳过对应行。
11. 完全空白行会被忽略。

前端验证：

1. 客户列表页能下载模板。
2. 客户列表页能选择 `.xlsx` 文件上传。
3. 导入完成后客户列表刷新。
4. 上传失败时显示后端返回的错误信息。

## 待确认点

1. 更新已有客户或联系人时，Excel 中的空白非必填字段是否允许覆盖系统原值为空。当前设计是允许覆盖。
2. 客户名称是否必须精确匹配才更新客户或关联联系人。当前设计是精确匹配，避免误更新。
3. 同一客户下同名联系人是否视为同一联系人并更新。当前设计是更新。
4. 导入结果错误明细本次是否只返回给接口，不做前端弹窗详情。当前设计是先不做详情弹窗。
